# Agent Memory Intro

# 全局架构

三层记忆 + 工具层：根据通用 Agent 框架（参考李航综述）和业界实践，Agent 信息存储可以分为以下层次

```
┌──────────────────────────────────────────────────┐
│               Agent Orchestrator (LLM)           │
├──────────┬───────────┬──────────┬────────────────┤
│ 短期记忆  │  长期记忆   │ 结构化知识 │   工具注册表    │
│ (Working │ (Long-Term │ (Knowledge│  (Tool        │
│  Memory) │  Memory)   │  Graph)   │   Registry)   │
└──────────┴───────────┴──────────┴────────────────┘
```

# 短期记忆 Working Memory

短期记忆维持当前会话连贯性。

短期记忆的管理本质是在**信息保真度**与**上下文窗口限制**之间做权衡。即使 100K+ token 窗口的模型，塞满上下文也会增加延迟和成本，仍需要主动管理。

| 时机 | 操作 | 说明 |
| --- | --- | --- |
| **每轮对话** | 追加 (Append) | 用户输入 + Agent 回复写入上下文缓冲区 |
| **工具调用返回** | 追加结果 | 工具执行的结果注入当前上下文 |
| **上下文逼近窗口上限** | 压缩 (Compress) | 触发摘要、滑动窗口裁剪或 token 级截断 |
| **会话结束** | 清空或归档 | 将有价值信息固化到长期记忆后释放 |

有三种主流策略：

1. **滑动窗口缓冲（Sliding Window Buffer）**
   - 保留最近 k 轮对话，最简单但有遗忘问题
   - 适合短交互场景（客服、简单问答）
2. **摘要缓冲（Summary Buffer）**
   - 当对话超过阈值时，用 LLM 对旧对话生成摘要，替换原始记录
   - **有损压缩**—— 牺牲细节换取更长跨度
   - LangChain 的 `ConversationSummaryBufferMemory` 即此模式
3. **Token 级缓冲（Token-Based Buffer）**
   - 按 token 数精确控制，确保不超出模型上下文窗口
   - 最精细但需要 tokenizer 配合

# 长期记忆 Long-Term Memory

让 Agent 能够**跨会话**记住信息 —— 用户偏好、历史交互、学习到的知识。

| 时机 | 操作 | 触发条件 |
| --- | --- | --- |
| **会话结束时** | 固化 (Consolidation) | 从短期记忆中提取关键事实，向量化后写入持久存储 |
| **任务完成时** | 经验存储 | 成功 / 失败的执行路径、反思结论写入记忆 |
| **显式指令** | 用户要求记住 | 请记住我喜欢用中文回答 |
| **周期性反思** | 记忆整理 | 定时触发 LLM 对记忆流做 reflection，生成高层洞察 |
| **检索时** | 按需读取 | 新会话开始时，根据用户查询语义检索相关记忆 |
| **知识过期** | 衰减 / 更新 | 旧记忆权重降低或被新信息覆盖 |

**写入流程（Consolidation Pipeline）：**

```
短期记忆 → LLM 提取关键事实 → Embedding 向量化 → 写入 Vector DB
                                                    ↓
                                          元数据标注（时间戳、来源、置信度）
```

**读取流程（Retrieval Pipeline）：**

```
用户查询 → Embedding → 相似度检索 Top-K → 重排序 → 注入短期记忆上下文
```

**三种长期记忆类型**（借鉴认知科学 Atkinson-Shiffrin 模型）：

| 类型 | 存储内容 | 实现方式 | 示例 |
| --- | --- | --- | --- |
| **情景记忆 (Episodic)** | 具体事件和交互经历 | 带时间戳的向量存储 | 上周用户问了 scaling law 的问题 |
| **语义记忆 (Semantic)** | 提炼后的知识和事实 | 知识图谱 / 结构化存储 | 用户是存储方向的工程师 |
| **程序记忆 (Procedural)** | 成功的执行模式和技能 | 代码 / 技能库 | Voyager 的技能库、M3-Agent 的策略 |

这三种类型在 Hang Li 论文的 **M3-Agent** 中得到了很好的体现 —— 它以实体为中心构建多模态记忆图，包含情景记忆（观察到的事件）和语义记忆（提取的知识片段）。

**记忆衰减与一致性：**

- **智能衰减**：根据访问频率、时间距离、重要性对记忆赋权，低权记忆被逐渐淡化
- **冲突检测**：新写入的记忆需与已有记忆做一致性校验，矛盾时以新信息为准或标记待确认
- **这也是当前最大痛点之一**—— 随着记忆规模增长，检索噪声增大，过时记忆可能导致错误

# 结构化知识 Knowledge Graph

将"记得"升级为"理解"。这也对应了 Hang Li 论文中强调的**神经符号处理**——Agent 的长期记忆中应包含符号处理的输出，以实体和概念为中心构建世界知识图谱。

| 维度 | 向量长期记忆 | 结构化知识图谱 |
| --- | --- | --- |
| 存储形式 | 文本嵌入 | 实体 + 关系 + 规则 |
| 查询方式 | 语义相似度 | 图遍历 + 逻辑推理 |
| 多跳推理 | 不支持 | 原生支持 |
| 确定性 | 概率性，有变异 | 确定性，可追溯 |
| 适用场景 | 模糊匹配、偏好召回 | 合规、分析、因果推理 |

# 工具注册表 Tool Registry

工具是 Agent 能力的**外延**，使 Agent 区别于纯 LLM。

| 生命周期阶段 | 操作 | 说明 |
| --- | --- | --- |
| **注册** | 工具声明 + Schema 定义 | 启动时或动态注册，声明名称、描述、参数 Schema、权限 |
| **发现** | 按需检索 | Agent 根据当前任务语义匹配可用工具（类似 ToolFormer） |
| **调用** | 执行 + 结果回注 | LLM 生成工具调用指令 → 执行器执行 → 结果写回短期记忆 |
| **更新 / 版本迭代** | 热更新 | API 变更、新工具上线时动态更新注册表 |
| **淘汰** | 下线 / 废弃 | 工具不可用时从注册表移除，避免 Agent 调用失败 |
| **学习** | 经验积累 | 记录工具使用成功率、延迟等指标，优化工具选择策略 |

**工具注册表（Tool Registry）示例：**

```yaml
Tool Registry
├── tool_name:  web_search 
│   ├── description:  搜索互联网获取实时信息 
│   ├── parameters: {query: string, max_results: int}
│   ├── auth: API_KEY
│   └── metadata: {avg_latency: 800ms, success_rate: 99.2%}
├── tool_name:  code_executor 
│   └── ...
└── tool_name:  calculator 
    └── ...
```

**MCP 协议（Model Context Protocol）**

MCP 是当前工具管理的**事实标准**，由 Anthropic 提出，已被 OpenAI、Google 等采纳。它的核心思想是：

- **标准化接口**：统一工具描述、调用、返回的协议格式（类比 AI 领域的 HTTP）
- **Server/Client 模型**：每个工具（或工具集）是一个 MCP Server，Agent 作为 MCP Client 连接
- **动态发现**：Agent 可以在运行时发现和连接新的 MCP Server
- **安全沙箱**：工具执行在隔离环境中，防止恶意行为

# 交互和场景

```
用户输入
   │
   ▼
[短期记忆] ← 加载当前上下文
   │
   ▼
[长期记忆检索] ← 语义匹配历史知识
   │
   ▼
[LLM 推理] → 决策：直接回答 / 调用工具 / 存储记忆 / 反思
   │                    │              │            │
   │              [工具注册表]     [长期记忆写入]  [记忆整理]
   │                    │
   │              [工具执行]
   │                    │
   ▼                    ▼
[短期记忆更新] ← 合并工具结果
   │
   ▼
输出回复
   │
   ▼
[会话结束?] → Yes → [固化关键信息到长期记忆]
```

**短期记忆解决"当下连贯"，长期记忆解决"跨会话积累"，结构化知识解决"可靠推理"，工具解决"能力外延"**。四者协同才能构建真正智能的 Agent 系统。

| 场景 | 推荐配置 |
| --- | --- |
| 简单单轮问答 | 短期记忆（滑动窗口）+ 基础工具 |
| 多轮对话助手 | 短期记忆（摘要缓冲）+ 长期记忆（向量存储）+ 工具集 |
| 个人助手 / 长期陪伴 | 三层记忆全配 + 情景 / 语义记忆 + 记忆反思机制 |
| 机器人 / 具身智能体 | 三层记忆 + 多模态记忆（M3-Agent 模式）+ 程序记忆（技能库） |
| 企业级严肃场景 | 三层记忆 + 知识图谱（结构化知识层）+ MCP 工具网关 + 审计日志 |

# Ref

1. **李航综述**
   https://jcst.ict.ac.cn/article/doi/10.1007/s11390-025-5951-5

> 机器人智能体M3-Agent由MLLM、多模态编码器、工具和多模态长期记忆组成。系统通过两个并行过程运行：记忆化和控制。在记忆化过程中，它持续接收来自环境的多模态（视频和音频）输入，以在其多模态记忆中创建和更新世界知识；在控制过程中，它根据外部指令基于记忆中存储的世界知识进行推理并提供回复。
>
> 在记忆化过程中，M3-Agent类似于人类，构建和更新**情景记忆**和**语义记忆**。前者记录观察到的事件，后者记录提取的知识片段，形式可以是文本、视频或音频。其最突出的特征是它构建以实体为中心的多模态记忆，形成对环境的理解，这构成了世界知识。例如，一个人的多模态信息（面孔、声音等）、一个人的属性（姓名、性别等）、人与人之间的关系，以及人与物之间的关系，都以多模态形式表示在图中。
>
> M3-Agent使用多模态工具（人脸识别和说话人识别）从长期记忆中检索相关知识，生成情景和语义记忆，并在图中形成新的节点和边；以上所有操作都是通过执行MLLM记忆化策略模型生成的命令来实现的。该策略模型通过使用半自动方法合成的数据进行模仿学习训练。
>
> 在控制过程中，M3-Agent根据指令从长期记忆中检索相关知识，进行多轮推理和检索，最终生成回复；这些操作也是通过执行MLLM控制策略模型生成的命令来完成的。该策略模型使用强化学习和标注数据进行训练。M3-Agent与当前RAG的区别在于它支持多轮推理和检索，而RAG通常是单轮的。
>
> M3-Agent在在线长视频问答任务中与代表性MLLM相比取得了显著更高的准确率。这是因为回答此类问题需要处理长视频中跨模态、跨实体和跨事件的信息，以及对场景的理解和推理；M3-Agent在这方面与传统方法相比具有更大的优势。
